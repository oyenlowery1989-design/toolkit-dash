## Tiered Rewards
- Route: `app/(tools)/tiered-rewards/page.tsx`
- Panel: `components/tiered-rewards/TieredRewardsPanel.tsx`, plus `TierBuilder.tsx` (tier list editor), `TierConfigCard.tsx` (per-config card: tiers, assets, JSON import, history, preview/run)
- Lib: `lib/tiered-rewards/{types,calculator,fetcher,runner,scheduler}.ts`
- Hook: `hooks/use-tiered-reward-configs.ts` — DB-backed cache, dual-mode (Supabase/SQLite)
- API: `/api/db/tiered-rewards` (config/tier/asset CRUD), `/api/tiered-rewards/{run,history}`
- Tiered per-holder reward distribution: define balance tiers (min inclusive / max exclusive, first-match-wins), attach one or more flat per-holder reward assets per tier; run manually (Preview then Execute, or direct "Run Now") or on a node-cron schedule (`lib/tiered-rewards/scheduler.ts`, skipped on Vercel)
- `calculatePreview()` fetches current holders via Horizon (paginated with retry/backoff), assigns to tiers, computes per-asset cost against sender's live balances (issuer-is-unlimited special case, fee-multiplier-scaled fee budget), blocks the run with explicit reasons if short or missing a trustline
- `runConfig()` sends batched (≤100 ops/tx, automatic bad-seq retry + per-op no-trust fallback) or one-at-a-time; every attempt logged to `run_log`, read by both the preview modal and per-config History tab
- Persistence dual-mode (SQLite locally, Supabase when `isSupabaseOnly()`) — CRUD route, run route, and scheduler each maintain their own load/write split
- Same-process execution-lock: `runConfig()` checks/adds to a globalized `_tieredRewardsRunningConfigs` Set before running (returns an error if the same config is already in flight) and wraps the actual run in `withAccountLock()` from `lib/stellar-submit.ts`; the scheduler's cron tick reads the same Set for a cheap early skip
- Partial PATCH to a config never wipes `secret_key`/schedule fields that weren't part of the patch payload — both SQLite and Supabase branches of the config "update" action only touch fields explicitly present in the request body

### reviewed+fixed 2026-07-13
- **CRITICAL — "Import & Replace" tiers previously deleted all existing tiers then silently failed to create any replacements.** The import path pre-generates a client-side `id` for what's actually a brand-new tier (so it can attach child assets immediately); `upsertTier` used to decide create-vs-update purely by `tier.id ? "update" : "create"`, so it always fired `"update"` against a non-existent row (silent no-op on SQLite, 404 on Supabase) — tiers gone, replacements never persisted, a run against that config would pay nobody. Fixed: `upsertTier` now decides the action by actual cache membership (checked before the optimistic mutation), and returns its write promise so `handleImportConfirm` can `await` each tier before creating its assets — also closes a latent tier-vs-asset creation race.
- **Scheduler dual-mode writes**: `loadEnabledConfigs()` already correctly branched on `isSupabaseOnly()` for reads, but the cron tick's `last_failure_at` writes and the `previewOnly` run-log insert were unconditional SQLite — mirrored `runner.ts`'s already-correct write pattern (`insertLogRows`, now exported; a new `setConfigLastFailure()` helper) into `scheduler.ts`'s 5 write sites.
- Batch-mode no-trust per-op retry: a holder in the unfinished tail of a batch (after an individual retry failed for a non-no-trust reason and the loop aborted) previously got zero run-log row at all — now logged as `"aborted"` before the abort propagates, so every holder in a real-money run has an audit row.
- Tier "update" API action now runs the same overlap validation "create" already enforced — editing an existing tier's min/max could previously introduce silent overlapping ranges, making `assignHoldersToTiers()`'s first-match-wins assignment iteration-order-dependent for borderline holders.
- `TierBuilder`'s `handleAddTier` now derives the next tierNumber/position from `Math.max(...)` of existing values, not from the current tier count — deleting a non-last tier then adding a new one no longer collides with a surviving tier's number/position.
- `lib/tiered-rewards/runner.ts`'s `extractError()` is now a thin wrapper around `getErrorMessage()` from `lib/stellar-helpers.ts` instead of a hand-rolled duplicate; `getOpResultCodes()` (separate — extracts the ops array for no-trust retry routing) is unchanged.
- **Deferred, not fixed this pass**: `shortAddr()` is used instead of `<ShortAddress>` throughout this module's components (TierBuilder/TierConfigCard/TierPreviewModal) — cosmetic, real layout-regression risk in the same pass as money-safety fixes; same gap exists in Auto-Send Groups.
