## Payments
- Route: `app/(tools)/payments/page.tsx` — single file (~2500 lines), no separate components
- **4 tabs**: Send, Path, Claimable Balance, Fee Bump
- Uses shared `useHorizonServer()` hook everywhere (no inline `new Horizon.Server(...)`); all confirm-dialog address rows use `<ShortAddress>` (Send leg destination, Path destination, Claimable claimants) — not raw address strings
- **Send tab**: multi-leg (asset + amount + destination per leg); wallet picker per leg; address book; ShortAddress badge on valid destinations
  - **Max button**: reserve+sponsorship-aware via `nativeMaxSpendable()` helper (reuses `lib/stellar-reserve.ts`'s `calcAvailableXlm`, fed `numSponsoring`/`numSponsored` now tracked on `AccountBalance`), minus an `estOps`-scaled tx fee estimate — not the old flat-reserve calc. Known ceiling: doesn't count cancel-ops for removeTrustline legs (`ponytail:` comment at the helper)
  - **Remove Trustline** (non-native only): now a `<Switch>` (was raw checkbox); on check → auto-fills max balance; amber warning if amount < full balance; preflight offers fetch is hoisted OUT of the per-leg loop (fetched once, not once per leg) and paginates past 200 via `.next()`; op order per leg: [cancel ops…] → payment → changeTrust(limit=0); `legCancelCountsRef` tracks cancel counts for correct op-index error mapping
  - Switching the signing account resets `legs[].amount`/`removeTrustline` and claim amount/asset (balance-derived state) — destinations are left alone, matching my-wallet's precedent
  - `submittingRef` guards `handleSubmit` against a double-invocation race (rapid double-click on Confirm & Submit); dialog's submit button also has `disabled={isSubmitting}`
- **Path tab**: strict-receive (exact dest, max send) and strict-send (exact send, min receive) toggle; calls `strictReceivePaths` / `strictSendPaths`; builds `pathPaymentStrictReceive` / `pathPaymentStrictSend`
  - Selecting a path prefills `maxSendAmount`/`minReceiveAmount` from that path's own quote; amber warning if the typed value is still worse than the selected quote (warning only, not blocking)
  - `pathsFetchedAt` timestamp + live 15s tick while the confirm dialog is open → amber "quote is over a minute old" warning in the dialog if stale
  - Confirm dialog shows the selected path's actual route (hop chain) + quoted amounts, not just typed form fields
  - Submit-catch has a path-specific branch for `op_over_source_max`/`op_under_dest_min`/`op_too_few_offers` → "re-run Find Paths" message (previously fell through to the Send-only recovery branches with a generic message)
  - Native-asset toggles are `<Switch>`, mode toggle + path-result cards are `<Button>` (was raw `<input type="checkbox">`/`<button>`)
- **Claimable Balance tab**: asset picker + amount + N claimants (all unconditional); `Operation.createClaimableBalance`
  - `isFormValid` enforces the protocol's 10-claimant cap and no-duplicate-destination check (Add button disables at 10, per-row duplicate warning) — previously only caught via a Horizon error after signing
  - Max button (same `nativeMaxSpendable` helper, `extraReserveXlm = 0.5 × claimant count`) + amber over-balance warning; static reserve-cost line ("locks N XLM of additional reserve")
- **Fee Bump tab**: paste inner XDR → live parse (op count + fee + signature count); `TransactionBuilder.buildFeeBumpTransaction(pubKey, baseFee, innerTx, networkPassphrase)`; Memo/Fee cards hidden on this tab
  - `innerPerOpFee` (derived from parsed fee/ops) is now enforced in `isFormValid` — typing a base fee below the inner tx's own per-op rate blocks submit with an inline message, instead of only failing at `buildFeeBumpTransaction`'s own throw
  - 0-signature inner tx flagged amber ("will fail with tx_bad_auth"); permanent amber reminder that XDR carries no network identifier — verify the inner tx matches the selected network
- **Trustline recovery**: detects op_no_trust with correct legOpIndex mapping (accounts for cancel + changeTrust ops); prompts "Add trustline & retry"; shows live status during retry
- **Error messages**: `getErrorMessage` in `lib/stellar-helpers.ts` extracts `result_codes` from Horizon 400 → "tx: tx_failed | ops: op_underfunded"
- **Layout density** (2026-07-13): tightened to fit ~1 screen at 1440×900 without scrolling — Memo+Fee merged into one card (was two separate Cards with their own headers), Send tab's Asset+Amount side-by-side instead of stacked, tighter Card header/content padding throughout, asset-balance status bar hidden entirely instead of showing a redundant empty-state message. Measured screen-fit ratios (viewport heights needed): Send 1.02, Fee Bump 1.00, Claimable 1.13, Path 1.35 (two asset fieldsets keep it the tallest tab — further compaction would mean merging each fieldset's Native-toggle row with its amount row, not done here, real risk to the non-native code/issuer layout for marginal gain)
